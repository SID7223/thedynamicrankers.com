# DATABASE ARCHITECTURE BLUEPRINT

Messaging, Tasks, Permissions, and Global Chat

This blueprint defines the core database structure for the platform's internal dashboard, messaging system, and task collaboration system. It is optimized for Cloudflare D1 (SQLite), high-concurrency messaging, and future AI integration.

The architecture supports:
- Threaded conversations
- Task-based discussions
- Global chat
- Search indexing (FTS5)
- Permission control
- Media attachments

---

## 1. Core Principles & ID Strategy
- **Primary Keys**: Every table uses a `TEXT PRIMARY KEY` with UUIDs (version 4) generated by the application layer. This ensures cross-environment compatibility and prevents ID enumeration.
- **Relational Integrity**: Foreign keys must be strictly enforced. `ON DELETE CASCADE` is used for child records (attachments, reactions) but restricted for primary records (users, tasks).
- **Search**: Leveraging SQLite's **FTS5 (Full-Text Search)** module for high-performance keyword indexing without needing an external search engine.
- **Time**: All timestamps are stored in `DATETIME` format as **UTC**. Conversion to Pakistan Standard Time (PKT) or user-specific timezones is handled at the presentation layer.

## 2. SQL Schema Definition (Cloudflare D1)

```sql
-- 1. Users Table
CREATE TABLE users (
    id TEXT PRIMARY KEY, -- UUID
    name TEXT NOT NULL,
    email TEXT UNIQUE NOT NULL,
    role TEXT CHECK(role IN ('admin', 'superuser', 'operative', 'viewer')) DEFAULT 'operative',
    timezone_preference TEXT DEFAULT 'Asia/Karachi',
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    updated_at DATETIME DEFAULT CURRENT_TIMESTAMP
);

-- 2. Tasks Table
CREATE TABLE tasks (
    id TEXT PRIMARY KEY, -- UUID
    task_number TEXT UNIQUE NOT NULL, -- e.g., TASK-101
    title TEXT NOT NULL,
    description TEXT,
    status TEXT CHECK(status IN ('backlog', 'todo', 'in_progress', 'review', 'done')) DEFAULT 'todo',
    created_by TEXT NOT NULL,
    assigned_to TEXT,
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    updated_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (created_by) REFERENCES users(id),
    FOREIGN KEY (assigned_to) REFERENCES users(id)
);

-- 3. Task Permissions Table (Extended Access)
CREATE TABLE task_permissions (
    id TEXT PRIMARY KEY,
    task_id TEXT NOT NULL,
    user_id TEXT NOT NULL,
    permission_level TEXT CHECK(permission_level IN ('viewer', 'contributor', 'admin')) DEFAULT 'contributor',
    granted_by TEXT NOT NULL,
    granted_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (task_id) REFERENCES tasks(id) ON DELETE CASCADE,
    FOREIGN KEY (user_id) REFERENCES users(id),
    FOREIGN KEY (granted_by) REFERENCES users(id),
    UNIQUE(task_id, user_id)
);

-- 4. Chat Rooms Table
CREATE TABLE chat_rooms (
    id TEXT PRIMARY KEY, -- UUID
    type TEXT CHECK(type IN ('task', 'global', 'direct')) NOT NULL,
    task_id TEXT UNIQUE, -- Null if global/direct
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (task_id) REFERENCES tasks(id) ON DELETE CASCADE
);

-- 5. Messages Table
CREATE TABLE messages (
    id TEXT PRIMARY KEY, -- UUID
    room_id TEXT NOT NULL,
    sender_id TEXT NOT NULL,
    parent_message_id TEXT, -- For threading
    message_content TEXT NOT NULL,
    message_type TEXT CHECK(message_type IN ('text', 'image', 'audio', 'gif', 'emoji', 'system')) DEFAULT 'text',
    edited BOOLEAN DEFAULT FALSE,
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    updated_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (room_id) REFERENCES chat_rooms(id) ON DELETE CASCADE,
    FOREIGN KEY (sender_id) REFERENCES users(id),
    FOREIGN KEY (parent_message_id) REFERENCES messages(id) ON DELETE CASCADE
);

-- 6. Message Attachments Table
CREATE TABLE message_attachments (
    id TEXT PRIMARY KEY,
    message_id TEXT NOT NULL,
    file_url TEXT NOT NULL,
    file_type TEXT NOT NULL,
    file_size INTEGER NOT NULL, -- in bytes
    file_name TEXT NOT NULL,
    uploaded_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (message_id) REFERENCES messages(id) ON DELETE CASCADE
);

-- 7. Message Mentions Table
CREATE TABLE message_mentions (
    id TEXT PRIMARY KEY,
    message_id TEXT NOT NULL,
    mentioned_user_id TEXT NOT NULL,
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (message_id) REFERENCES messages(id) ON DELETE CASCADE,
    FOREIGN KEY (mentioned_user_id) REFERENCES users(id)
);

-- 8. Notifications Table
CREATE TABLE notifications (
    id TEXT PRIMARY KEY,
    user_id TEXT NOT NULL,
    type TEXT NOT NULL, -- task_access, mention, reply, status_change
    reference_id TEXT, -- ID of the related task or message
    message TEXT NOT NULL,
    read_status BOOLEAN DEFAULT FALSE,
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (user_id) REFERENCES users(id) ON DELETE CASCADE
);

-- 9. Task Access Requests Table
CREATE TABLE task_access_requests (
    id TEXT PRIMARY KEY,
    task_id TEXT NOT NULL,
    requesting_user_id TEXT NOT NULL,
    status TEXT CHECK(status IN ('pending', 'approved', 'rejected')) DEFAULT 'pending',
    approved_by TEXT,
    requested_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    approved_at DATETIME,
    FOREIGN KEY (task_id) REFERENCES tasks(id) ON DELETE CASCADE,
    FOREIGN KEY (requesting_user_id) REFERENCES users(id),
    FOREIGN KEY (approved_by) REFERENCES users(id)
);
```

## 3. Optimized Search Indexing (FTS5)

To support instant global search across millions of messages, we utilize SQLite FTS5 virtual tables.

```sql
-- Create FTS5 virtual table for messages
CREATE VIRTUAL TABLE message_search_index USING fts5(
    message_id UNINDEXED,
    room_id UNINDEXED,
    content
);

-- Trigger to keep search index updated automatically
CREATE TRIGGER after_message_insert AFTER INSERT ON messages BEGIN
    INSERT INTO message_search_index(message_id, room_id, content)
    VALUES (new.id, new.room_id, new.message_content);
END;

CREATE TRIGGER after_message_update AFTER UPDATE ON messages BEGIN
    UPDATE message_search_index SET content = new.message_content
    WHERE message_id = old.id;
END;

CREATE TRIGGER after_message_delete AFTER DELETE ON messages BEGIN
    DELETE FROM message_search_index WHERE message_id = old.id;
END;
```

## 4. API Endpoint Specifications

### **Messaging API**
| Method | Path | Description | Body Params |
| :--- | :--- | :--- | :--- |
| `GET` | `/api/chat/rooms` | List accessible rooms | - |
| `GET` | `/api/chat/rooms/:id/messages` | Fetch messages for a room | `cursor`, `limit` |
| `POST` | `/api/chat/messages` | Send a new message | `room_id`, `content`, `parent_message_id`, `attachments[]` |
| `PATCH` | `/api/chat/messages/:id` | Edit a message | `content` |
| `GET` | `/api/chat/search` | Full-text search | `q` (query string) |

### **Tasks & Permissions API**
| Method | Path | Description | Body Params |
| :--- | :--- | :--- | :--- |
| `POST` | `/api/tasks` | Create new task | `title`, `description`, `assigned_to` |
| `POST` | `/api/tasks/:id/access-request` | Request access to a task | `reason` |
| `PATCH` | `/api/tasks/:id/permissions` | Update user permission | `user_id`, `level` |

## 5. Data Validation Rules & Constraints

1.  **Users**:
    - `email`: Must match standard email regex.
    - `role`: System must have at least one `admin`.
2.  **Tasks**:
    - `task_number`: Automated format `TASK-` + incremental counter.
    - `status`: Transition rules (e.g., cannot move from `backlog` to `done` directly).
3.  **Messages**:
    - `message_content`: Maximum length 4000 characters.
    - `parent_message_id`: Must belong to the same `room_id`.
4.  **Attachments**:
    - `file_size`: Maximum 25MB per file for D1-linked storage metadata.
    - `file_type`: Restricted to `['image/*', 'audio/*', 'application/pdf', 'video/*']`.

## 6. Routing and URL Structure

- **Tasks**: `/dashboard/tasks/:task_number`
- **Global Chat**: `/dashboard/chat/global`
- **Task Specific Chat**: `/dashboard/tasks/:task_number/chat`
- **Threaded View**: `/dashboard/chat/message/:message_id`
- **Search Results**: `/dashboard/search?q=keywords`

---

## 7. Future AI Compatibility
- **Contextual Ingestion**: AI can query the `messages` table filtered by `room_id` to understand the history of a specific task.
- **Thread Analysis**: By following `parent_message_id`, AI agents can reconstruct decision-making trees.
- **Automated Summarization**: Structured `task_number` and `message_type` fields allow AI to focus on text for status reports.

---

## 8. Migration Rules
Any change to this architecture requires updating the **`SCHEMA_NOTICE.txt`** and providing the corresponding **`ALTER TABLE`** or **`CREATE TABLE`** commands in a timestamped migration file.
